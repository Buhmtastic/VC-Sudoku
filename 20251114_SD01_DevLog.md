## ğŸ“ Key Design Decisions

### 1. Strict OOP Adherence
**Decision**: All attributes MUST be private (underscore prefix)

**Rationale**:
- Enforces encapsulation
- Prevents external code from breaking invariants
- Demonstrates professional OOP knowledge
- Prepares for future refactoring

**Impact**: More verbose, but cleaner architecture

---

### 2. Dependency Injection Over Hard-Coding
**Decision**: Board and Solver receive Validator via injection

**Rationale**:
- Loose coupling
- Easy testing (can inject mock validators)
- Follows Dependency Inversion Principle
- Flexible composition

**Example**:
```python
validator = Validator()
board = Board()
board.set_validator(validator)  # Injection
```

---

### 3. Property Pattern for Read-Only Access
**Decision**: Use `@property` for all read-only attributes

**Rationale**:
- Pythonic way to control access
- Can add validation/computation later
- Clean syntax: `cell.value` instead of `cell.get_value()`
- Maintains encapsulation

---

### 4. Comprehensive Documentation
**Decision**: Every class and public method has detailed docstring

**Rationale**:
- Serves as living documentation
- Helps with IDE autocomplete
- Demonstrates professionalism
- Useful for portfolio review

**Format**: Google-style docstrings
